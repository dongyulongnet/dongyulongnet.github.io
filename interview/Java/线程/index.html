Java 线程

线程间的通信

wait/notify

都是Object类的native方法。
wait会释放锁并暂停当前线程，将机会交给其他线程。
需要synchronized代码块内或方法内使用,否则会报异常java.lang.IllegalMonitorStateException。

await/signal

都是java.util.concurrent.locks.Condition类的方法。
需要配合java.util.concurrent.locks.ReentrantLock的lock块使用。
多个线程可以争夺一把锁，并做到精确线程的通知。

CountDownLatch

AQS是对类AbstractQueuedSynchronizer缩写
CountDownLatch内部维护了count,通过构造函数传入。
可以通过countDown()函数扣减count,当count变为0时会唤醒等待线程，等待需要由CountDownLatch触发。

volatile

  • 保证可见性
    volatile修饰的变量写之后将本地内存刷新到主内存，保证了可见性
  • 保证有序性
    volatile变量读写前后插入内存屏障以避免重排序，保证了有序性
  • 不保证原子性

经典的DCL单例模式属性是通过volatile标记的。

CAS

CAS是compare and swap缩写，多线程下可以实现同步机制，基于CPU的原子指令cmpxchg,不会造成所谓的数据不一致问题。
如Java中Unsafe提供的方法compareAndSwapInt底层实现即为CPU指令cmpxchg。

CAS原理

CAS 的实现离不开处理器的支持。如果程序是在多处理器上运行，就为 cmpxchg 指令加上 lock 前缀（lock cmpxchg）。反之，如果程序是在单处理器上运行，就省略 lock 前缀（单处理器自身会维护单处理器内的顺序一致性，不需要 lock 前缀提供的内存屏障效果）。如果我们仔细分析 concurrent 包的源代码实现，会发现一个通用化的实现模式：

  • 首先，声明共享变量为 volatile；
  • 然后，使用 CAS 的原子条件更新来实现线程之间的同步；
  • 同时，配合以 volatile 的读 / 写和 CAS 所具有的 volatile 读和写的内存语义来实现线程之间的通信。

堆中对象份分配

再给一个对象分配内存的时候不是原子性的操作，至少需要以下几步：查找空闲列表、分配内存、修改空闲列表等等，这是不安全的。解决并发时的安全问题也有两种策略：

  • CAS：实际上虚拟机采用CAS配合上失败重试的方式保证更新操作的原子性，原理和上面讲的一样。
  • TLAB：如果使用CAS其实对性能还是会有影响的，所以 JVM 又提出了一种更高级的优化策略：每个线程在 Java 堆中预先分配一小块内存，称为本地线程分配缓冲区（TLAB），线程内部需要分配内存时直接在 TLAB 上分配就行，避免了线程冲突。只有当缓冲区的内存用光需要重新分配内存的时候才会进行CAS操作分配更大的内存空间。

虚拟机是否使用TLAB，可以通过-XX:+/-UseTLAB参数来进行配置（jdk5及以后的版本默认是启用TLAB的）

CAS引发的问题

ABA问题

多个线程同时执行CAS操作，以T1、T2两个线程为例，执行顺序如下：

  • T1线程执行读操作，获取原值A，此时线程执行权被切换
  • T2线程执行CAS操作，将A值修改为B
  • T2线程再次执行CAS操作,将B值修改为A
  • 此时T1线程恢复执行，将比较值与原值相同，然后将新值写入内存，完成CAS操作

对于 ABA 问题，通常的处理措施是对每一次 CAS 操作设置版本号。
从Java1.5开始JDK的atomic包里提供了一个类AtomicStampedReference来解决ABA问题。这个类的compareAndSet方法作用是首先检查当前引用是否等于预期引用，并且当前标志是否等于预期标志，如果全部相等，则以原子方式将该引用和该标志的值设置为给定的更新值。

循环时间长开销大

自旋CAS（不成功，就一直循环执行，直到成功） 如果长时间不成功，会给 CPU 带来非常大的执行开销。

wait底层实现

synchorized代码块加锁时会执行monitorenter指令，获取对象的monitor。
而进行wait之前需要先与monitor建立联系，这也是为什么wait必须在同步代码块中使用的原因。
多核环境下，即使多个线程同时执行monitorenter指令，也只会有一个线程会与monitor建立联系。
当开始执行wait时,会将当前线程放在wait set中,并放弃lock对象上的所有声明，直至被唤醒。
notify会选择wait set中任意一个线程唤醒，notifyAll会唤醒wait set中所有线程。
被唤醒的线程需要重新竞争锁对象，获得锁的线程可以从wait处继续往下执行。

多线程应用

创建线程的方式

线程执行任务的接口类有Runnable和Callable,分别通过对方法run()和call()方法的调用来执任务，其中Callable有返回值。
而创建线程的方式主要通过Thread类来完成。

  • 继承Thread类
  • 实现Runnable接口，通过Tread构造参数形式传入
  • 实现Callable接口，通过FutureTask构造参数包装后，再传入Tread构造函数

最终都是通过Thread类的start方法来启动新线程执行任务

线程池的能力支持

线程池通过FutureTask构造函数包装来返回数据。
通过重构了三个submit方法，来执行Runnable和Callable类型的任务

Spring的能力支持

Spring通过AsyncResult构造函数的包装来返回数据

线程数估算

根据CPU核心数确定线程池并发线程数

Nthreads=Ncpu*(1+w/c)

  • IO密集型

一般情况下，如果存在IO，那么肯定w/c>1（阻塞耗时一般都是计算耗时的很多倍）,但是需要考虑系统内存有限（每开启一个线程都需要内存空间），这里需要上服务器测试具体多少个线程数适合（CPU占比、线程数、总耗时、内存消耗）。如果不想去测试，保守点取1即，Nthreads=Ncpu*(1+1)=2Ncpu。这样设置一般都OK。

IO密集型=2Ncpu（可以测试后自己控制大小，2Ncpu一般没问题）（常出现于线程中：数据库数据交互、文件上传下载、网络数据传输等等）

  • 计算密集型：

假设没有等待w=0，则W/C=0. Nthreads=Ncpu。计算密集型=Ncpu（常出现于线程中：复杂算法）

线程调度策略

Java使用的线程调度是抢占式调度，在JVM中体现为让可运行池中优先级高的线程拥有CPU使用权，如果可运行池中线程优先级一样则随机选择线程，但要注意的是实际上一个绝对时间点只有一个线程在运行（这里是相对于一个CPU来说，如果你的机器是多核的还是可能多个线程同时运行的），直到此线程进入非可运行状态或另一个具有更高优先级的线程进入可运行线程池，才会使之让出CPU的使用权，更高优先级的线程抢占了优先级低的线程的CPU。

放弃CPU使用权时机

  • 主动放弃CPU
    JVM暂时放弃CPU操作（基于时间片轮转调度的JVM操作系统不会让线程永久放弃CPU，或者说放弃本次时间片的执行权）例如调用yield()方法。
  • 因为某些原因进入阻塞状态
    例如阻塞在I/O上。
  • 当前运行线程结束，即运行完run()方法里面的任务。

调度机制

Java的线程的调度机制都由JVM实现。Java把线程优先级分成10个级别，线程被创建时如果没有明确声明则使用默认优先级，JVM将根据每个线程的优先级分配执行时间的概率。有三个常量Thread.MIN_PRIORITY(1)、Thread.NORM_PRIORITY(5)、Thread.MAX_PRIORITY(10)分别表示最小优先级值（1）、默认优先级值（5）、最大优先级值（10）。
线程的调度策略决定上层多线程运行机制，JVM的线程调度器实现了抢占式调度，每条线程执行的时间由它分配管理，它将按照线程优先级的建议对线程执行的时间进行分配，优先级越高，可能得到CPU的时间则越长。

七个核心参数

参数
解释
含义
corePoolSize 核心线程大小 corePoolSize 是线程池中的一个最小的线程数量，即使这些线程处理空闲状态，他们也不会被销毁，除非设置了allowCoreThreadTimeOut。

线程锁

ReentrantLock和Synchronized

  • 使用方式
    Synchronized可以修饰实例方法，静态方法，代码块。自动释放锁。
    ReentrantLock一般需要try catch finally语句，在try中获取锁，在finally释放锁。需要手动释放锁。
  • 实现方式
    Synchronized是重量级锁。重量级锁需要将线程从内核态和用户态来回切换。耗费系统资源.
    ReentrantLock是轻量级锁。采用cas+volatile管理线程，不需要线程切换，获取锁线程觉得自己肯定能成功，这是一种乐观的思想（可能失败）。
    前者是被动阻塞悲观锁，状态是block，后者是主动的阻塞乐观锁，状态是wait。
  • 公平和非公平
    Synchronized只有非公平锁。
    ReentrantLock提供公平和非公平两种锁，默认是非公平的。公平锁通过构造函数传递true表示。
  • 可重入锁
    Synchronized和ReentrantLock都是可重入的，Synchronized是本地方法是C++实现，而ReentrantLock是JUC包用Java实现。
    在ReentrantLock中，重入次数用整形state表示。进入1次递增1次，出来1次递减1次。
  • 可中断的
    Synchronized是不可中断的。
    ReentrantLock提供可中断和不可中断两种方式。其中lockInterruptibly方法表示可中断，lock方法表示不可中断。
  • 条件队列
    Synchronized只有一个等待队列。
    ReentrantLock中一把锁可以对应多个条件队列。通过newCondition表示。
